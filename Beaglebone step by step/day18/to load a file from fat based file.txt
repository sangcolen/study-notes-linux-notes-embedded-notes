to load a file from fat based file system in to memory use: fatload
to load a file from any file system: load
example:
fatload usb 0:1 0x82000000 uImage
fatload mmc 0:1 0x88000000 initramfs
load mmc 0:1 0x88000000 uImage

👉fatload⁠ (Lệnh đặc định hệ thống tập tin - Filesystem-specific Command):
 Đây là lệnh thế hệ cũ. Nó ép U-Boot chỉ được phép đọc dữ liệu từ phân vùng định dạng FAT (FAT12, FAT16, FAT32).
 Nếu mày dùng ⁠fatload⁠ trên phân vùng định dạng ⁠ext4⁠ (như cái phân vùng ⁠1:1⁠ trên eMMC của mày), U-Boot sẽ báo lỗi 
 ngay lập tức vì driver FAT không hiểu được cấu trúc dữ liệu của EXT4.
👉load⁠ (Lệnh vạn năng - Generic Filesystem Command):
 Đây là lệnh thế hệ mới của U-Boot. Nó tích hợp một tầng trừu tượng (Abstraction Layer) tự động quét qua các driver Filesystem có sẵn (FAT, EXT2/3/4, BTRFS, CramFS...).
 Khi mày gõ ⁠load⁠, U-Boot tự mò mẫm xem cái phân vùng mày chỉ định (⁠1:1⁠ hay ⁠0:1⁠) đang xài định dạng gì để tự lôi driver tương ứng ra đọc.

1. Dùng lệnh ls generic để quét file
lệnh này để xem danh sách file ở thư mục gốc hoặc thư mục boot:

Plaintext
=> ls mmc 1:1 /
Hoặc:

Plaintext
=> ls mmc 1:1 /boot
Nếu lệnh này chạy được, nó sẽ show ra một đống file. Có file nào tên là zImage, vmlinuz-xxx hay uImage không nhé.

2. Dùng lệnh load generic thay thế
Nếu thấy file Kernel (ví dụ nó nằm ở /boot/vmlinuz-5.10.x-ti-rxx hoặc đại loại thế), sẽ dùng chính lệnh load để nạp:

Plaintext
=> load mmc 1:1 0x82000000 /boot/tên_file_kernel_mày_vừa_thấy

1. Tại sao lúc đầu lỗi?
Thứ nhất: Trên con eMMC này, hệ điều hành Debian mặc định đéo dùng file uImage (định dạng ảnh boot cũ của U-Boot), mà nó dùng file vmlinuz-4.19.94-ti-r42 (nhìn dòng cuối cùng ở hình image_9fe587.png kìa). File vmlinuz bản chất là một Kernel dạng nén (zImage), nên mày bắt buộc phải dùng lệnh bootz chứ đéo được dùng lệnh bootm.

Thứ hai: File kernel của mày nó nằm ở đường dẫn /boot/vmlinuz-4.19.94-ti-r42 chứ đéo nằm ở thư mục gốc /.

2. Gõ lệnh thực hành chuẩn chỉnh ngay lập tức cho tao:
Bây giờ mày gõ đúng combo 3 lệnh này theo thứ tự để nạp và boot thử từ eMMC:

Bước 1: Load file Kernel (vmlinuz) vào RAM
Mày dùng lệnh load generic nạp file này vào địa chỉ RAM quen thuộc:

Plaintext
=> load mmc 1:1 0x82000000 /boot/vmlinuz-4.19.94-ti-r42
(Chỗ này nó phải báo thành công và in ra dòng: 10095592 bytes read... thì mới ăn tiền).

Bước 2: Muốn boot được Linux trên chip ARM hiện đại, mày cần nạp thêm file Device Tree (.dtb)
Hồi nãy ở hình image_9fe587.png mày thấy có thư mục <DIR> dtbs đúng không? Cái file .dtb cho con BeagleBone Black nó nằm trong đó. Mày nạp nó vào một địa chỉ RAM khác (ví dụ 0x88000000) bằng lệnh:

Plaintext
=> load mmc 1:1 0x88000000 /boot/dtbs/4.19.94-ti-r42/am335x-boneblack.dtb
(Nếu gõ lệnh trên mà báo lỗi không tìm thấy file, mày có thể gõ ls mmc 1:1 /boot/dtbs/4.19.94-ti-r42/ trước để check xem chính xác tên file .dtb của con Black là gì nhé).

Bước 3: Kích hoạt Boot bằng lệnh bootz
Vì đây là file vmlinuz (zImage) nên cấu trúc lệnh boot sẽ là: bootz [Địa_chỉ_Kernel] [Địa_chỉ_Initrd] [Địa_chỉ_Device_Tree]. Ở đây mình chưa cần dùng initrd tạm thời thì bỏ trống nó bằng dấu gạch ngang -:

Plaintext
=> bootz 0x82000000 - 0x88000000

ERROR DIFFERENT🚫🚫🚫🚫
Bên khóa học dùng ⁠bootm⁠ với file ⁠uImage⁠: Định dạng ⁠uImage⁠ có một cái Header (đoạn mã mồi) chứa metadata bọc bên ngoài Kernel. 
Khi dùng lệnh ⁠bootm⁠, chính bản thân U-Boot sẽ đọc cái Header này trước khi nhảy vào Kernel. U-Boot phát hiện ra cấu hình file này 
yêu cầu phải có FDT (Flattened Device Tree) hoặc ATAGS nhưng mày đéo truyền vào, nên U-Boot chủ động quăng ra câu chửi 
rồi ra lệnh reset một cách "lịch sự".
 Mày dùng ⁠bootz⁠ với file ⁠vmlinuz⁠: Lệnh ⁠bootz⁠ nhảy thẳng vào thực thi nhân Linux dạng nén. Hệ thống đéo có tầng kiểm tra trung gian nào của U-Boot cản lại nữa. 
Kernel mù thông tin phần cứng \rightarrow Chết đứng \rightarrow Watchdog đập reset.
